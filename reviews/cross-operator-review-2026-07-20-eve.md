## Verdict: PASS (with one code fix, committed alongside this review)

## Finding 1 — receiver never exits at server EOF (FIXED in this commit)

**Symptom:** when the server stops or restarts, the receiver hangs forever, silent —
the user must Ctrl-C and rerun without knowing why. This amplifies the already-known
"receiver restart → re-tune" weakness: the receiver can't even *notice* the restart.

**Root cause:** OpenBSD `nc` (the flavor Termux installs as `netcat-openbsd`) stays
alive after network EOF as long as its local stdin is open — and in the pipeline
`nc | tail | mpv`, nc's stdin is the terminal. Network closes, nc keeps running,
the pipe stays open, mpv waits at a dry pipe.

**Evidence (executed, pasted not paraphrased):**
- Stock pipeline, 3 s stream: killed by a 10 s timeout (`exit: 124`). Reduced case
  `nc 127.0.0.1 48100 | tail -c +17 | wc -c` also hung → the hang is nc, not mpv.
- With `-d` (don't read stdin): `576000` bytes delivered, `exit: 0` —
  byte-exact (3 s × 48000 Hz × 2 ch × 2 B = 576,000; header stripped correctly).
- Full fixed pipeline with real mpv playback: exit 0 at ~4.5 s wall (≈ 3 s audio
  + ~1 s connect/buffer). Post-fix, the actual `bedcast-receive.sh` was re-run
  as shipped: exit 0 at 4.2 s.

**Fix:** `nc -d` in `receiver/bedcast-receive.sh` and the README "any other
receiver" one-liner. (`-d` exists in OpenBSD nc; other flavors vary — the
traditional-nc fallback path in a future receiver should re-check.)

## Finding 2 — the operator's live ~2.5 s desync is v0 behaving as documented

Not a bug: v0's sync-by-buffer-depth trade, pre-declared in README and
WIRE-FORMAT. Accounting: receiver `--audio-buffer` 0.5 s + `--cache-secs` 0.5 s
+ demuxer readahead + TCP socket buffers both sides + WASAPI event buffering
plausibly sums to seconds. Remedy is the documented one: set the PC player's
audio delay ≈ **−2.5 s** once (VLC: Tools → Track Synchronization, or `j` held;
each press is −50 ms). Holds until pause/seek — exactly the wound the v1
timestamp spec closes. No change requested.

## Finding 3 — undocumented single-client serialization (should-fix, docs only)

`Program.cs` accepts one client and blocks in `ServeClient` until it dies; a second
phone connecting queues **silently** in the OS backlog — no error, no audio — until
the first drops. `server-linux.sh` honestly prints "one client at a time"; the
Windows server and README say nothing. One sentence in the README fixes the
surprise.

## Finding 4 — ghost-client window (minor, LAN-acceptable)

Dead-client detection is `Poll(readable) && Available == 0` plus write-path
`IOException`. A client that vanishes without RST (phone WiFi sleep) triggers
neither promptly; the server can hold a ghost for minutes of TCP retransmit
while a reconnecting phone waits in the backlog (compounding Finding 3). Worth a
code comment; fine for the stated trust model.

## Nit

- README says "mpv: `+`/`-`" for audio-offset compensation; mpv's default binds
  are Ctrl + `+` / Ctrl + `-` (100 ms steps). Unverified by this review — check
  before editing.
